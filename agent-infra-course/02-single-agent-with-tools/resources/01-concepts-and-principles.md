# 02 单 Agent 与 Tools 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

## 1. 单 agent 的核心，不是“会调用函数”
很多人第一次做 agent，会把 function calling 当成全部。但真正的单 agent 系统至少有 4 个层次：
- 工具契约：模型能调用什么
- 决策点：何时调用哪个工具
- 执行点：工具如何真正运行
- observation：运行结果如何回到系统

所谓 agent loop，本质上就是“根据当前状态决定动作，再根据动作结果更新状态”的循环。

## 2. 为什么 tool schema 比 prompt 更值得先做
prompt 可以影响模型行为，但 schema 决定的是系统的确定性边界。schema 不清晰时，你会看到：
- 模型不知道什么时候该选哪个工具
- 参数经常缺失或格式错误
- 失败返回结构不一致
- trace 难以统一分析

所以在工程上，通常要先收紧 schema，再微调描述和 prompt。

## 3. tool calling 和普通函数调用的区别
普通函数调用是程序员自己决定参数和执行路径；tool calling 是模型先生成一个结构化动作，再由系统去执行。因此它多了一层“不确定的动作生成”。你要做的工程治理，就是尽量把这层不确定性收拢到可控范围内。

## 4. 什么时候其实不需要 agent
如果流程步骤固定、工具选择固定、错误处理路径固定，那么更适合 workflow，而不是自由 agent。单 agent 最适合的是“存在局部开放性决策，但又不需要把整个系统都放开”的场景。
