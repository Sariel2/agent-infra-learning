# 02 单 Agent 与 Tools 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一篇的目标，是把“tool calling”从表面功能，讲成一个真正的系统能力。很多人第一次接触 agent 时，会觉得这一层只是“让模型会调函数”。但工程上最重要的，恰恰不是“能不能调起来”，而是“调用过程是否稳定、可解释、可治理”。

## 1. 单 agent 的核心，不是“会调用函数”
如果只从表面看，tool calling 很像“模型替你决定调用哪个函数”。但一个真正可用的单 agent 系统，至少包含 4 个层次：
- 工具契约层：系统允许模型使用哪些能力
- 决策层：模型如何在多个工具之间做选择
- 执行层：工具如何真正运行，以及如何处理异常
- observation 层：工具结果如何回到 agent，成为下一步决策输入

这四层里，只要有一层没设计好，就会出现典型问题：
- 工具选错
- 参数填错
- 执行失败但系统不知怎么处理
- 工具结果返回了，但后续系统消费不了

所以你要把 agent 想成“模型动作治理系统”，而不是“函数调用增强器”。

## 2. 为什么 tool schema 比 prompt 更值得先做
很多新人遇到工具调用不稳定，第一反应是继续改 prompt。prompt 当然重要，但在工程顺序上，schema 常常更优先。

原因很直接：prompt 主要影响模型偏好，schema 决定系统边界。

当 schema 不清楚时，系统会出现这些问题：
- 模型不知道某个工具的适用条件
- 参数字段含义模糊，生成经常缺项
- 成功和失败返回结构不统一
- 后续 trace 和 eval 很难稳定分析

换句话说，schema 是在告诉模型和系统：
- 这项能力叫什么
- 什么时候该用
- 需要哪些输入
- 成功后会得到什么
- 失败后会得到什么

如果这些约束没有被表达清楚，后面再怎么润 prompt，稳定性也很难真正上来。

## 3. tool calling 和普通函数调用到底差在哪
这是一个非常值得想清楚的点。

### 普通函数调用
是开发者自己决定：
- 调哪个函数
- 传什么参数
- 在什么时机调用

### tool calling
则变成：
- 模型先生成一个结构化动作
- 运行时解释这个动作
- 系统执行工具
- 系统再把结果喂回模型或上层逻辑

这意味着，多出了一整层不确定性：
- 动作可能选错
- 参数可能不完整
- 工具可能超时或失败
- 工具结果可能不符合后续预期

所以 agent infra 的重点，根本不是“多一个工具接口”，而是治理这层由模型生成动作带来的不确定性。

## 4. observation 为什么重要
很多人只盯着“工具有没有调用成功”，但 observation 真正的价值在于：它是下一步决策的输入。

一个好的 observation 不只是原始返回值，而是经过统一包装后的状态材料。它通常要回答：
- 这次工具调用是否成功
- 成功时产出了什么关键信息
- 失败时失败类型是什么
- 是否值得重试
- 是否应该换工具或直接终止

如果 observation 只是杂乱文本，agent 后续就很难稳定判断下一步。也就是说，observation 不是“日志”，而是 agent 状态流的一部分。

## 5. 为什么工具返回统一结构比想象中更重要
如果每个工具各自返回不同形状的数据，系统很快会退化：
- 每个调用点都要写单独解析逻辑
- trace 聚合越来越混乱
- fallback 逻辑散落在各处
- eval 很难比较不同工具路径

因此课程里一直强调统一的 `ToolResult` 风格，不是因为追求“写法整齐”，而是因为没有统一返回结构，就没有后面的可观测、可测试、可评测。

统一结构通常至少要包含：
- `success`
- `data`
- `error`
- `metadata`

这四类字段不是固定写法，但它们代表的是一种重要原则：系统要能稳定地区分正常路径和异常路径。

## 6. 单 agent 最适合的到底是什么问题
单 agent 最适合的，不是“所有智能任务”，而是这种问题结构：
- 有多个可选工具
- 工具选择存在局部不确定性
- 每次动作后的 observation 会影响下一步
- 但整体任务规模还没大到需要多角色拆分

比如：
- 客服问题先判断是知识问答、订单查询还是工单创建
- 查完订单状态后，再决定是否需要补一条解释
- 先搜文档，再根据命中结果决定是否追问用户

这些任务都具有“局部开放性 + 总体可控”的特点，非常适合单 agent。

## 7. 什么时候其实不需要 agent
这是这模块最重要的判断题之一。

如果一个需求满足以下特点：
- 步骤基本固定
- 工具选择基本固定
- 分支路径基本可枚举
- 人工审核点非常明确

那更适合 workflow，而不是 agent。因为 agent 带来的自由度也是成本：
- 难测
- 难回归
- 难解释
- 难合规

所以你不是“能用 agent 就一定要用”，而是要先判断开放性到底值不值得引入。

## 8. 为什么课程坚持先单 agent，再谈 multi-agent
因为单 agent 已经足够暴露 agent 系统里最核心的问题：
- schema 是否清晰
- observation 是否结构化
- trace 是否统一
- fallback 是否存在
- 工具失败时系统是否还能收束

如果这些基础治理能力都还没有建立，直接上 multi-agent，通常只会把复杂度翻倍。多主体系统不会自动解决单主体系统的问题，只会把它们分散到更多角色和更多 trace 里。

## 9. 常见误区
### 误区一：只要能跑起来，就说明工具设计没问题
不是。能跑只是最低标准。真正的问题是：是否可解释、可回归、可扩展。

### 误区二：工具描述越长越好
也不是。描述应该足够清楚，但过长会让重点模糊。最关键的是明确边界和参数含义。

### 误区三：失败重试就能解决大部分问题
只有部分问题适合重试。选错工具时，重试可能只是重复错误；参数本身不合法时，重试也没意义。要先判断失败类型。

### 误区四：tool calling 就等于 agent
很多系统只是“模型触发工具 + 一次性返回”，这更像带工具的推理链。只有当 observation 真正进入后续决策时，agent 味道才会更强。

## 10. 学完本篇的完成标准
你应该能稳定回答这些问题：
- 为什么 schema 要比 prompt 更早治理
- 为什么 observation 不是普通返回值
- 为什么工具结果必须统一结构
- 什么情况下该用单 agent，什么情况下不该用
- 单 agent 的复杂度边界在哪里

只要你能把这些问题讲清楚，这一模块的底层认知就建立起来了。后面的代码和服务，只是在把这些原则运行化。
