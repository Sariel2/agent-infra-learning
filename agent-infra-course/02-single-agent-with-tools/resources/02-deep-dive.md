# 02 单 Agent 与 Tools 细节深挖

这一篇从实现角度往下拆。上一节回答的是“为什么要这么设计”，这一节回答的是“这些设计在代码里通常表现成什么结构，错误又会从哪里冒出来”。

## 1. 最小 agent loop 到底在做什么
课程中的最小 loop 可以拆成 5 个步骤：
1. 接收用户问题
2. 识别当前问题最可能需要哪种能力
3. 生成工具动作和参数
4. 执行工具，并把结果整理成 observation
5. 根据 observation 结束、重试或进入下一轮

即使当前示例只做了一轮，它已经足够帮你看清 agent 的骨架。多轮 agent 只是把这条链路重复几次，并在 observation 基础上继续推进，而不是一个完全不同的新系统。

## 2. 为什么 trace 结构要一开始就统一
你后面真正会经常问的问题，不是“能不能跑”，而是：
- 这次到底调用了哪个工具
- 参数是什么
- 为什么参数是这样
- 工具结果是什么
- 失败发生在哪一层
- 是否触发了重试
- 整个过程花了多久

如果这些信息没有被结构化记录，调试会迅速退化成“看控制台输出猜问题”。这就是为什么课程里的 `ToolTrace` 或同类对象虽然简单，但它代表着一条重要工程原则：过程必须可被稳定记录。

## 3. tool calling 的三类失败不能混为一谈
### 第一类：选错工具
症状通常是：
- 用户问订单问题，却去查知识库
- 明明该建工单，却只返回一段解释

这类问题往往出在：
- 工具描述不清
- 上游分类逻辑模糊
- prompt 没把边界讲明

### 第二类：参数错误
症状通常是：
- 缺少必要字段
- 参数格式不合法
- 枚举值不在预期范围

这类问题通常说明：
- schema 太松
- 参数校验不足
- 模型缺少清晰的参数语义提示

### 第三类：工具执行失败
症状通常是：
- 超时
- 下游服务异常
- 返回结构和预期不符

这类问题修复重点不在模型，而在工具层、运行时包装层和超时重试策略。

如果把三类失败都概括成“agent 不稳定”，你就会找错修复点。

## 4. 为什么统一的 ToolResult 很关键
如果每个工具都返回不同形状的数据，系统很快会出现这些问题：
- 成功路径解析逻辑重复
- 失败路径缺乏统一处理方式
- trace 聚合时很难对比
- fallback 逻辑到处散落

所以统一的 `ToolResult` 不是形式主义，而是后续治理能力的前提。它让你可以把失败类型显式化，把 metadata 沉淀下来，并把上游逻辑从具体工具细节中解耦出来。

## 5. timeout、retry、fallback 为什么必须分开看
很多初学者会把这三件事混成一句“失败了就重试一下”。实际上它们分别解决不同问题。

### timeout
防止某个工具无限阻塞整个系统。它是在控制时间边界。

### retry
适合处理短暂性失败，比如临时网络波动。它是在处理瞬时不稳定。

### fallback
适合在主要路径失败后，切换到更保守、更稳定的替代方案。它是在控制业务连续性。

举个例子：
- 工具参数本身无效
这类问题不适合 retry，但可能适合 fallback 到“请用户补全信息”。

所以设计恢复策略时，一定要先区分失败类型，而不是统一套重试。

## 6. 反例：把所有事情都交给模型
一个典型错误是，明明某些步骤完全可以固定，却把所有控制权都交给模型。看起来很灵活，实际后果通常是：
- 波动大
- 回归难
- 出错后不知该看哪层
- 一次 prompt 调整影响多个路径

这也是 agent infra 中非常重要的一个判断能力：
- 哪些地方值得保留自由度
- 哪些地方应该用工程结构收紧不确定性

比如：
- 工具参数校验，应该更偏工程结构
- 工具选择边界，可以允许部分模型判断
- 异常返回包装，应该严格结构化

## 7. support-agent 为什么是这一模块的收束点
`support-agent` 的价值不只是把三个工具拼起来，而是让你第一次真正看到这些东西如何组合成一个小型运行时：
- tool schema
- agent loop
- result wrapper
- trace
- timeout / retry / fallback

也就是说，这个模块真正学的不是“我会写一个会调工具的 demo”，而是“我知道怎样把一个 demo 往稳定系统方向推进一步”。

## 8. 怎么判断示例代码写得是不是对
你不需要一开始就追求最复杂的 agent。更重要的是看这些问题：
- 工具边界是否单一清晰
- 参数是否有明确约束
- 成功和失败是否都能被上游消费
- trace 是否能还原执行过程
- 出错后系统是否能收束，而不是直接崩掉

只要这些点成立，即使实现很小，它也已经具有正确的工程方向。

## 9. 本模块和后续模块的连接点
这一模块是后续三件事的前提：
- 第 3 模块会把工具调用扩展到 retrieval 和 memory
- 第 4 模块会把 loop 和 trace 纳入更完整的 workflow runtime
- 第 5 模块会把这里的单 agent 抽象成 pattern 进行比较

所以如果这里没把 schema、observation、trace 的概念想透，后面会一路模糊。

## 10. 学完本篇的完成标准
你应该能自己拆解一次工具调用失败，并明确判断它属于：
- 决策问题
- 参数问题
- 执行问题

同时你也应该能解释：
- 为什么统一 trace 很重要
- 为什么 `ToolResult` 应该稳定存在
- 为什么 timeout、retry、fallback 不是一回事

一旦你能做到这几点，你就不再只是“会接一个工具”，而是真正开始理解 agent runtime 的底层结构。
