# 03 RAG、Memory 与 Knowledge 细节深挖

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：01-concepts-and-principles.md](./01-concepts-and-principles.md) | [下一篇：03-code-reading-guide.md](./03-code-reading-guide.md)

这一篇把 RAG 系统常见问题进一步拆开。核心目标不是让你“知道更多术语”，而是让你在系统效果不好时，能把问题拆到正确层级，而不是一句“RAG 效果不好”就结束。

## 1. RAG 的三个故障层次
一个知识问答系统出错，通常来自三层：
- 检索层：没找到对的片段，或者排序错误
- 拼接层：上下文组织方式不合理，导致关键信息被淹没
- 回答层：即使给了正确上下文，答案综合仍然跑偏

这三层看起来最后都表现成“答案不对”，但修复方式完全不同。

### 检索层问题
常见症状：
- 根本没命中相关片段
- 命中了，但排序靠后
- 命中很多相似片段，却没有最关键的一条

### 拼接层问题
常见症状：
- 候选片段里其实有答案，但被无关上下文淹没
- 多个片段之间顺序不合理，模型抓错重点
- chunk 太长或太短，导致模型无法稳定综合

### 回答层问题
常见症状：
- 给了正确上下文，但模型综合错误
- 答案主题接近，却细节失真
- 正文回答和 citation 指向不一致

工程上最重要的一点是：不要把三层问题混成一句“RAG 效果不好”。

## 2. 为什么课程示例先用最朴素的检索
因为它能把“检索命中”这件事单独显影出来。你可以直接观察：
- query token 和 doc token 的重叠关系
- score 是如何变化的
- citation 最终来自哪个文档

如果一上来就把检索做成黑盒，你往往只能看到结果对不对，却不知道命中链路到底发生了什么。

这也是课程设计上的一个重要原则：
- 先让系统透明
- 再逐渐把底层实现升级

## 3. chunking 真正影响的不是文本长度，而是知识边界
chunking 看似只是“把文档切开”，但它真正决定的是：
- 哪些语义单元会被一起召回
- 哪些信息会被拆散
- citation 能否稳定落在合理范围

可以把 chunk 看成“检索时最小知识单元”。如果切法不合理，后面再好的检索算法也只能在错误粒度上工作。

### chunk 太小的问题
- 语义被打散
- 检索命中了局部词，但缺少上下文
- 回答时需要更多片段拼接，增加上下文污染风险

### chunk 太大的问题
- 单片段里混入太多无关信息
- citation 指向不够精确
- 相似主题容易彼此干扰

所以 chunking 本质上是在平衡“召回粒度”和“上下文纯度”。

## 4. citation mismatch 为什么是单独一类错误
有一种很常见的情况：系统看起来答得差不多，但 citation 对不上。这说明问题不是简单的“没命中”，而是：
- 检索阶段其实命中了
- 回答阶段抓错了片段
- 或者 citation 在接口组装时被错误绑定

这类问题非常重要，因为它会制造一种危险幻觉：
- 正文看似合理
- 但系统已经失去可审计性

如果你不专门监控 citation consistency，这种错误会长期潜伏。

## 5. memory 的三个典型失败模式
### 无界累积
上下文越来越长，最后模型失去聚焦能力，成本也越来越高。

### 错误信息被长期保留
如果系统把错误回答、错误推断也一起记进 memory，后面会不断放大污染。

### 记住了不该记住的东西
并不是所有历史都值得进入未来推理。很多内容只对某一轮问题有意义，过了这一轮就应该被丢弃或摘要。

这也是为什么 memory 设计不能只问“记不记”，还要问：
- 记多久
- 记什么
- 用什么形式记
- 什么时候删

## 6. 知识更新为什么是一个系统问题
很多示例只演示“导入文档再提问”，但真实知识系统还要面对：
- 文档新增
- 老文档替换
- 文档冲突
- 版本回退
- 不同资料可信度不同

如果没有版本意识，系统很容易出现这种现象：
- 老资料和新资料同时被召回
- 模型把过期资料当成最新结论
- citation 明明存在，但来源版本已经不可靠

所以 knowledge system 的治理对象，不只是“检索”，还包括知识生命周期。

## 7. hosted 与 self-hosted 的真正区别
很多人把它们简单理解成“一个省事，一个可控”。这当然没错，但还不够具体。

### hosted retrieval 更像平台能力
你把更多复杂度交给平台，换来更快的集成速度和更低的起步门槛。

### self-hosted retrieval 更像基础设施能力
你自己承担更多索引、召回、调优和运维责任，换来更强的可解释性和可控制度。

所以你选哪种方案，不只是技术偏好，而是在选择：
- 你想把复杂度放在哪
- 你当前更缺“速度”还是更缺“控制”

## 8. knowledge-agent 为什么是本模块的收束点
`knowledge-agent` 真正体现的不是“我做了个知识问答 API”，而是：
- 检索不再是孤立函数
- citation 成为响应结构的一部分
- memory 被视为有界状态，而不是无限日志
- 失败样本能被记录和复盘

这说明你开始从“能答问题”走向“能治理知识系统”。

## 9. 怎么判断 RAG 系统迭代方向是否正确
你可以用几个问题来约束自己：
- 我现在是在优化检索、拼接，还是回答阶段
- 这次改动有没有让 citation 更可靠
- memory 是否更有界、更干净
- 是否保留了足够的 trace，方便继续定位

如果这些问题答不清楚，就说明系统迭代仍然停留在“凭感觉调效果”阶段。

## 10. 学完本篇的完成标准
你应该能拿一个错误样例，把问题拆成：
- 检索没命中
- 拼接不合理
- 回答阶段综合失真
- citation 绑定错误
- memory 污染

只要你能稳定地这样拆问题，RAG 对你来说就不再是“神秘黑盒”，而是一条可以被治理的工程链路。
