## 1. RAG 的三个故障层次
一个知识问答系统出错，通常来自三层：
- 检索层：没找到对的片段，或者排序错误
- 拼接层：上下文拼接方式不合理，导致关键信息被淹没
- 回答层：即使给了正确上下文，答案综合仍然跑偏

工程上很重要的一点是：不要把三层问题混成一句“RAG 效果不好”。

## 2. 课程示例为什么先用最朴素的检索
因为它能把“检索命中”这件事单独显影出来。你可以直接观察：
- query token 和 doc token 的重叠程度
- score 如何变化
- citation 来自哪个文档

这比一开始就用复杂向量检索更适合入门，因为你能看清系统到底在做什么。

## 3. memory 为什么必须有上限
记忆如果无限增长，会导致：
- token 成本不断上升
- 无关上下文越来越多
- 模型越来越难分清当前问题的真正重点

所以课程示例把短期记忆设计成有界队列。你后面可以把它升级成摘要式记忆，但“有界”这个原则不会变。
