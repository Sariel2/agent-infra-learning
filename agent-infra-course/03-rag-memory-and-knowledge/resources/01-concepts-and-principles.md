# 03 RAG、Memory 与 Knowledge 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一篇要把 RAG 和 memory 从“热门名词”讲回系统能力。很多人学到这一块时，会把注意力全部放在向量库、embedding 或某个框架上。但真正影响系统质量的，往往是你有没有把 retrieval、memory、citation、知识更新这些边界分清楚。

## 1. RAG 不是“把文档塞进向量库”
RAG 真正要解决的问题是：当模型自身参数记忆不够，或者你必须依赖最新、私有、可审计的资料时，怎样把与当前问题最相关的外部知识，以可控方式引入上下文。

因此 RAG 的核心不是某个向量库，而是一条完整链路：
- 文档如何进入系统
- 文档如何被切成可检索单元
- 问题如何被转成检索请求
- 系统如何召回候选片段
- 片段如何被排序、裁剪、拼接
- 回答阶段如何使用这些片段
- 最终怎样保留 citation

如果你只盯着“向量库选哪个”，会错过真正决定质量的链路设计问题。

## 2. retrieval 与 memory 的边界
这是本模块最重要的边界之一。

### retrieval
面向外部知识：
- 文档
- 资料
- FAQ
- 私有知识库
- 最新事实

### memory
面向当前交互状态：
- 最近用户意图
- 当前任务进展
- 本轮对话上下文
- 需要短期延续的信息

可以用一句话区分：
- retrieval 解决“我需要查什么”
- memory 解决“我刚才说到了哪”

如果把两者混在一起，系统常见退化方式是：
- 把历史对话误当知识依据
- 把文档片段误当短期任务状态
- 回答时引用了聊天历史，却以为自己在引用知识库

## 3. citation 为什么是系统能力，而不是装饰能力
很多人把 citation 理解成“给回答加个来源，看起来更专业”。实际上 citation 的工程价值非常高，因为它能帮助你回答三个关键问题：
- 这条回答依据了哪份资料
- 检索到底命中了什么
- 如果回答错了，是检索错、拼接错，还是综合错

没有 citation，RAG 系统就像黑箱。你只能感觉“这次好像答对了”或“这次好像答偏了”，但几乎无法稳定定位问题层级。

所以 citation 不是美化输出，而是可审计性的一部分。

## 4. 为什么先从最小检索实现开始
课程先从本地最朴素检索入手，不是因为它最强，而是因为它最适合把链路显影出来。你能直接看清：
- query 和文档是如何匹配的
- score 是怎么来的
- citation 是怎么被保留下来的

如果一上来就用复杂的托管检索或向量数据库，虽然更接近真实环境，但初学时容易被黑盒感压住，看不到系统到底在哪一步做了什么。

## 5. chunking 真正影响的是什么
很多人把 chunking 当成“把长文档切成几段”。这是不够的。chunking 实际上决定了三个核心东西：
- 召回粒度
- 语义完整性
- citation 可解释性

### chunk 太小
- 召回看起来精确
- 但语义容易被打散
- 回答时上下文缺少完整句群或段落

### chunk 太大
- 语义更完整
- 但容易混入无关内容
- citation 难以精确指向

所以 chunking 本质上是在平衡“上下文纯度”和“语义完整度”，而不是简单切文本。

## 6. memory 为什么必须是有界的
新手很容易把 memory 理解成“把所有历史都塞回上下文”。这几乎一定会出问题：
- token 成本不断升高
- 无关上下文越来越多
- 模型越来越难聚焦当前问题
- 错误信息可能被长期污染保留

因此课程中的 memory 设计故意从有界窗口开始。你后面当然可以升级成摘要式记忆、事件记忆甚至用户画像，但“有界”和“可裁剪”通常是不会失效的原则。

## 7. hosted retrieval 与 self-hosted retrieval 的关系
课程把两者都讲，不是为了让你一次学会所有方案，而是为了让你理解它们分别解决什么阶段的问题。

### hosted retrieval
适合：
- 快速建立成功体验
- 先把整个 RAG 链路跑通
- 先把注意力放在系统集成，而不是底层检索细节

### self-hosted retrieval
适合：
- 需要更强的控制力
- 需要理解索引、chunk、召回、排序这些内部细节
- 需要做更细致的调优和排障

它们不是“谁高级谁低级”，而是控制力和学习目标不同。

## 8. knowledge system 和单次问答有什么不同
很多教程最后停留在“给一段文档，然后问一个问题”。这只能说明你做了一个单次检索示例，还不能说明你理解了 knowledge system。

真正的 knowledge system 还需要考虑：
- 文档如何导入
- 索引何时更新
- 老版本资料怎么替换
- 引用如何保留
- 失败样本如何记录
- 会话上下文如何与知识检索协同

也就是说，本模块的目标不是“会做一个检索 demo”，而是开始建立对知识系统生命周期的认识。

## 9. 常见误区
### 误区一：RAG 效果不好，就是向量库不行
很多时候问题根本不在向量库，而在 chunking、召回、排序、上下文拼接或回答阶段。

### 误区二：memory 越多越聪明
不是。memory 太多通常只会让噪音增多，模型更难聚焦。

### 误区三：citation 只是锦上添花
没有 citation，你几乎无法稳定排查系统到底在哪一步出了问题。

### 误区四：检索命中了，就说明系统没问题
检索命中只是第一步。后面还可能在上下文拼接、回答综合、引用绑定等阶段继续出错。

## 10. 学完本篇的完成标准
你应该能讲清：
- RAG 到底是一条什么链路
- retrieval 和 memory 的边界是什么
- citation 为什么是系统能力
- chunking 真正影响什么
- hosted 与 self-hosted retrieval 各自适合什么阶段

一旦这些边界建立起来，你后面看到任何 RAG 框架，都会更容易把它拆回这些基本问题，而不是被工具名称带着走。
