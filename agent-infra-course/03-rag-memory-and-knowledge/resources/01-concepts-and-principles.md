# 03 RAG、Memory 与 Knowledge 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

## 1. RAG 不是“把文档塞进向量库”
RAG 是一条完整的数据链路：
- 文档进入系统
- 文档被切分成可检索片段
- 问题被转成检索请求
- 系统取回候选片段
- 片段被送入回答阶段
- 最终答案最好附带 citation

所以 RAG 的关键不只是向量库，而是“上下文是如何被挑选并带着证据进入回答过程”的。

## 2. retrieval 与 memory 的边界
retrieval 解决的是“我需要哪些外部知识”；memory 解决的是“我当前对话里该保留哪些状态”。

具体来说：
- retrieval 面向知识库、文档、外部事实
- memory 面向最近对话、用户偏好、当前任务状态

如果把两者混在一起，你会看到一种常见坏现象：系统把历史对话误当知识，把知识片段误当会话状态。

## 3. citation 为什么是工程能力
citation 不是为了让回答看起来学术，而是为了让系统可审计、可调试。没有 citation，你很难回答：
- 这条答案是依据哪个片段给出的
- 检索到底有没有命中正确资料
- 回答错是因为没检索到，还是因为综合阶段跑偏

## 4. 为什么课程先做本地检索再谈 hosted / vector DB
因为你需要先理解链路，再换更复杂的底层实现。只要链路没搞清楚，向量库、rerank、embedding 参数这些都只会让问题更模糊。
