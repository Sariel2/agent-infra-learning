# 06 转型、简历与面试 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

很多人把“简历、表达、面试”看成项目做完后的包装层，好像只是最后润色一下。对 agent infra 这种跨原理、工程、系统设计、产品理解的方向来说，这个想法很危险。因为转型成功与否，往往不取决于你有没有做过项目，而取决于你能不能把项目解释成别人愿意相信的能力证据。

这一篇要解决的是：为什么表达不是附属技能，而是工程交付的一部分；以及一个技术项目在转型语境下，到底该被组织成什么样。

## 1. 为什么最后一模块不是“附属内容”
代码存在，不等于能力被看见。现实面试中，面试官看不到你整个学习过程，他只能通过这些载体认识你：
- 简历 bullet
- README
- 项目讲稿
- 系统设计说明
- 面试中的追问回答

如果这些载体没有把你的真实能力组织出来，前面花很多时间做的东西，最后很可能只在你自己脑子里成立。

所以这一模块的目标不是“写得好看一点”，而是把你的经验重新编码成别人可以快速消费的证据结构。

## 2. 什么叫“可被相信的项目表达”
一个可信的项目表达，通常同时具备 4 个维度：
- 目标明确：这个系统解决什么问题
- 结构清楚：关键组件怎么协作
- 失败真实：你遇到过什么问题、怎么定位
- 收益具体：最后改进了什么，哪怕是认知收益

如果只讲目标和结果，容易像空洞宣传。
如果只讲结构，不讲为什么这么设计，容易像背框架。
如果只讲失败，不讲收束，容易像问题清单。

真正强的表达，是把这四个维度连成一条因果链。

## 3. 为什么要同时准备业务版和 infra 版讲法
同一个项目，面对不同听众，问题焦点是完全不同的。

### 业务版
业务版关注的是：
- 系统解决了什么实际问题
- 提升了什么效率或准确性
- 为什么这个方案值得做
- 和旧方案相比有什么变化

### infra 版
infra 版关注的是：
- 为什么这样分层
- 为什么这里用 workflow / agent / multi-agent
- tool schema、memory、eval、trace 是怎么治理的
- 哪些地方存在复杂度权衡

如果你只会 infra 版，业务方会觉得你只会讲术语。
如果你只会业务版，技术面试官会觉得你只是在复述结果。

所以成熟表达不是“能讲清一个版本”，而是“能在不同上下文中切换口径，同时保持事实一致”。

## 4. 为什么失败案例反而会增强说服力
很多人害怕讲失败，担心这会削弱自己。事实上，失败恰恰是最能体现工程能力的地方。

因为失败案例天然包含一条完整的工程链路：
- 你如何发现异常
- 你如何缩小问题范围
- 你如何判断根因
- 你如何改设计或代码
- 你如何验证修复有效

在 agent infra 场景里，这条链尤其重要。因为这个方向本来就充满非确定性，真正有价值的不是“我一次做对了”，而是“我知道如何把不稳定系统逐步治理稳定”。

## 5. 项目表达为什么必须保留“取舍”而不只是“功能”
很多项目介绍会退化成：
- 做了 RAG
- 接了 tools
- 加了 memory
- 做了 tracing

这种表达的问题在于，它只是在报菜名，没有体现判断。

真正能帮助你转型的是下面这种表达：
- 为什么这里用 workflow 而不是自由 agent
- 为什么这里先做 hosted RAG，再做 self-hosted 对比
- 为什么 memory 要做有界裁剪，而不是无限保留
- 为什么 eval 指标要先定义成功率和 citation 质量

因为工程师真正的价值，不是“把功能拼上去”，而是“知道该怎么取舍”。

## 6. 为什么这一模块里仍然保留代码工具
像 `project-story-api`、`interview-qa-pack`、`showcase-pack` 这类小工具，目的不是炫技，而是把“表达”这件事也工程化。

这背后有两个原则：
- 好表达不是临场发挥，而是可复用结构
- 输出资产应该沉淀为系统，而不是只存在于脑子里

当你把项目故事、常见问答、系统图、README 模板做成一套结构化产物时，你实际上是在给自己的转型过程建立“可复用表达层”。

## 7. 技术转型里最容易犯的 4 个表达错误
### 错误一：只会讲框架名
这会让人感觉你在追热点，而不是在做系统设计。框架名应该服务于判断，不应该替代判断。

### 错误二：只讲成功，不讲失败
这样会让表达失去真实感，也会让人怀疑你是否真的做过深层排障。

### 错误三：只有一种口径
不会根据面试官视角调整表达，通常意味着你对项目理解还没有内化成结构。

### 错误四：把学习项目讲成生产系统
这是另一个常见风险。学习项目可以很有价值，但不能虚构上线规模、用户量和收益。更好的做法是诚实说明：
- 这是自建训练项目
- 它重点验证哪些工程问题
- 你从中学到了哪些真实判断

诚实不会削弱你，虚构才会。

## 8. 为什么“学习项目”仍然可以非常有含金量
很多人担心：不是线上项目，面试官会不会看不上。

关键不在于是不是线上，而在于项目是否体现了这些能力：
- 问题是否清晰
- 设计是否成体系
- 是否有失败案例和治理动作
- 是否有评测、trace、测试等工程证据
- 是否能解释取舍

一个结构完整、讲得清楚、能经得起追问的学习项目，通常比一个“参与过一点但讲不清”的真实项目更能说明能力。

## 9. 这一模块和前面 5 个模块是什么关系
前 5 个模块产出的，是能力本身：
- Python 工程化基础
- 单 agent 与 tools
- RAG 与 memory
- workflow、observability、eval
- agent pattern 与架构判断

第 6 模块做的，是把这些能力重新组织成可展示资产：
- README
- 项目故事
- 简历 bullet
- 系统设计图
- 面试问答包

所以它不是“额外内容”，而是把学习成果变成职业资产的收束模块。

## 10. 学完本篇的完成标准
你应该能做到：
- 用 3 分钟讲清一个项目的目标、方案、失败、收益
- 用 10 分钟展开讲系统结构和关键取舍
- 用业务版和 infra 版两种视角介绍同一项目
- 在被追问“为什么不用另一种方案”时，给出稳定回答
- 明确区分“我做过什么”和“我从中验证过什么”

如果你能做到这些，这一模块才算真的完成。因为转型的本质，不只是学会新技术，更是把自己重新表达成一个适合该岗位的人。
