# 06 转型、简历与面试 细节深挖

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：01-concepts-and-principles.md](./01-concepts-and-principles.md) | [下一篇：03-code-reading-guide.md](./03-code-reading-guide.md)

这一篇把“表达”进一步拆开，讲它在实际转型和面试里到底是如何工作的。换句话说，这里不只是讲你要准备什么材料，而是讲：面试官是怎么听的、他为什么会追问、你的材料为什么会失效，以及如何把这些材料组织成可复用系统。

## 1. 简历 bullet 和项目讲稿到底有什么不同
很多人误以为项目讲稿只是简历 bullet 的加长版。实际上它们是两种完全不同的输出形式。

### 简历 bullet 的任务
简历 bullet 要解决的是“在很短时间内被扫描并留下印象”。它强调：
- 高密度
- 信息压缩
- 关键词清晰
- 结果前置

它更像索引，而不是完整叙事。

### 项目讲稿的任务
项目讲稿要解决的是“在被追问时仍能站得住”。它强调：
- 逻辑链条完整
- 前因后果清楚
- 取舍理由明确
- 能承接进一步追问

所以这两者不是简单长短差异，而是面向不同阅读场景的两种结构。

## 2. 为什么很多技术人一到面试就“讲不清”
根因通常不是表达能力天生不好，而是没有把经历重构成可输出结构。常见表现包括：
- 想到哪讲到哪
- 只记得做过的功能，不记得为什么这样设计
- 被问失败案例时，脑子里没有整理过根因链
- 被问“为什么不用别的方案”时，只能回答“当时觉得这样比较合适”

这些问题都说明一件事：项目还停留在“做过”的层面，没有沉淀成“可讲述的经验”。

## 3. 好的项目叙事为什么必须包含失败链路
项目最有说服力的部分，通常不是“功能做成了”，而是你如何让一个不稳定系统逐渐变稳定。

一个高质量失败链路，通常应该能讲清 5 件事：
- 现象：最初看到的异常是什么
- 定位：你如何缩小问题范围
- 根因：最终确认的问题在哪一层
- 修复：你改了什么设计或实现
- 验证：你如何证明问题真的缓解了

例如：
- “RAG 回答不稳定”
这不是一个好失败表述，因为它太笼统。

更好的表达是：
- “我们发现 citation 经常和答案正文不一致。先通过 trace 拆分出检索、上下文拼接、回答三个阶段，最后定位到是 chunk 过大导致上下文里混入多段相似内容，模型在回答时抓了正确主题但引用错了片段。之后调整 chunk 粒度并补充 citation consistency check，误引率明显下降。”

这类表达会立刻让人感觉你做的是工程治理，而不是泛泛试错。

## 4. 面试中真正高频的不是“定义题”，而是“判断题”
你当然可能被问：
- 什么是 RAG
- 什么是 tool calling
- 什么是 ReAct

但真正区分水平的，往往是判断题：
- 为什么这里用 workflow 而不是 agent
- 为什么 memory 不能无限增长
- 为什么 tool schema 要严格定义
- 为什么这个系统不适合 multi-agent
- 为什么 eval 指标先做这些而不是那些

这些问题的难点不在于定义，而在于你是否有稳定判断原则。也正因此，课程前 5 个模块一直在训练“边界、失败模式、取舍”，就是为了让你在这里不至于只会背术语。

## 5. “项目包装”为什么不是造概念，而是做信息分层
很多人一听到包装，就担心是不是要夸大其词。这里的包装不是吹，而是把同一事实按不同层级组织出来。

一个项目至少应该分成这几层信息：
- 一句话版本：这是个什么系统，解决什么问题
- 一段话版本：核心流程和价值
- 3 分钟版本：目标、结构、难点、结果
- 10 分钟版本：架构、关键模块、失败案例、取舍
- 深挖版本：任意一个点都能展开到设计细节和代码证据

如果没有这几层分级，你很容易在不同场景下失衡：
- 写简历时写太多，重点不突出
- 面试时讲太少，撑不起追问
- 技术追问时只能重复概念，落不到系统细节

## 6. 为什么 showcase-pack 是必要的收束动作
没有 `showcase-pack`，前面 5 个模块的成果很容易散落成：
- 一堆例子代码
- 多个 README
- 若干实验记录
- 一些图和笔记

这些内容对学习有帮助，但对转型不够友好，因为它们缺少一个统一表达入口。

`showcase-pack` 的本质，是把分散资产重新编排成：
- 项目主页
- 系统图
- demo 入口
- 常见问答
- 简历 bullet
- 讲稿模板

这不是新技术开发，而是信息组织工程。它做得越好，你越能在短时间内把“我学过”转成“我能证明我掌握了”。

## 7. 如何把 Java 后端经验翻译成 agent infra 语言
这是转型里特别重要的一步。你不是要抹掉原来经历，而是要把已有能力映射到新方向。

例如：
- 你过去强调接口设计，现在可以映射到 tool schema、response schema、API contract
- 你过去强调分层架构，现在可以映射到 route、service、provider、runtime、eval pipeline
- 你过去做过可观测性，现在可以映射到 tracing、run metadata、error attribution
- 你过去做过缓存、队列、异步任务，现在可以映射到 agent workflow 的状态流与任务编排

这样做的价值是：你不是“从零转行”，而是在已有工程能力之上增加一个新领域。这会让你的叙事更稳，也更真实。

## 8. 反例：哪些表达方式会立刻削弱可信度
### 反例一：项目描述全是名词堆砌
例如“使用 LangGraph、OpenAI SDK、Qdrant、FastAPI 构建多智能体平台”。这句话信息量看似很大，但几乎没告诉别人你到底解决了什么问题。

### 反例二：夸大结果
如果项目是训练项目，就诚实说训练项目。不要虚构“上线服务了多少用户”“提升多少百分比”，除非你真有证据。

### 反例三：回避失败
一个项目完全没有失败和修复过程，通常会让人怀疑你讲的是包装稿，而不是实际经历。

### 反例四：把教程知识直接背成答案
如果回答听起来像文档复读，而不是自己的工程判断，面试官会很快追问到你暴露真实掌握程度。

## 9. 为什么这模块仍然需要代码
有些人会问，既然这是面试和转型模块，为什么还要保留代码工具和服务框架？

因为表达最怕失去证据。没有代码、没有测试、没有可运行 demo，你的项目故事会越来越抽象。保留代码有三个作用：
- 让 README 和讲稿有可回指的证据
- 让你在回答深挖问题时能落回具体实现
- 让“展示资产”不是纯文档，而是可运行样例

所以这里的代码不是重点，但它是表达可信度的重要支撑。

## 10. 学完这一篇的完成标准
你应该能够把任意一个模块项目做成下面这些输出：
- 一段适合写在简历里的高密度 bullet
- 一段适合 3 分钟讲清的项目介绍
- 一套适合技术面试追问的架构讲解
- 一条失败案例的完整定位与修复链
- 一份能把旧经验映射到 agent infra 的自我介绍

如果你能把这些输出稳定做出来，这一模块的价值就真正落地了。因为对转型来说，最终决定你能不能拿到机会的，往往不是你看过多少资料，而是你能不能把已学会的东西讲成有证据、有判断、有可信度的能力。
