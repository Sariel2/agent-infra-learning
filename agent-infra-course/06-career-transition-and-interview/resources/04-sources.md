# 06 转型、简历与面试 原始来源与出处详解

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：03-code-reading-guide.md](./03-code-reading-guide.md) | [下一篇：05-common-mistakes.md](./05-common-mistakes.md)

这一页的目标，不是给你补更多“技术内容”，而是帮你把前 5 个模块的工程实践，重新接回行业叙事和面试语言。第 06 模块最核心的任务，是把“我做过这些代码”升级成“我能把这些实践讲成有判断、有证据、有行业语境的项目故事”。

## 怎么使用这一页
- 先读完第 06 模块正文，再看这里。
- 看这些来源时，不要只想“我要在面试里引用哪个名词”，而要想“我要怎样把自己的项目放回主流技术语境里讲”。
- 关注叙事语言、系统边界和取舍表达，而不只是框架名称。

## 1. [OpenAI new tools for building agents](https://openai.com/index/new-tools-for-building-agents/)
- 为什么这个来源重要：它能帮助你把自己的项目表达放回当前行业主叙事里，而不是只停留在“我做了个个人 demo”。对转型来说，这一点非常重要，因为面试官通常不仅在看你做了什么，也在看你是否理解当前行业正在如何组织这类系统。
- 这个来源原本在解决什么问题：
  - 如何把模型、tools、runtime、tracing 等能力看成一条统一开发路径
  - 如何降低 agent 系统开发的碎片化程度
  - 如何给开发者一个更连贯的平台视角
- 你应该从原文里抓到的核心原理：
  - 你的项目不应该像孤立作品，而应该能接上行业主线。
  - 面试表达里真正重要的，不是堆框架名，而是说明你的项目正在解决行业里真实存在的问题结构。
  - 如果你能把自己的项目放回“工具、runtime、tracing、eval”这条主线中，表达会更自然、更可信。
- 本教程吸收成了哪些内容：
  - `为什么项目要能放回行业语境里讲`
  - `业务版 + infra 版` 双重表达
  - showcase-pack 中对项目故事主线的组织
- 读原文时重点看什么：
  - tools、agents、tracing 被如何统一叙述
  - 平台在强调什么问题结构
  - 你的项目和这条叙事可以怎样对应
- 读完后你应该能回答：
  - 为什么项目讲法不能只停在“我用了哪些框架”
  - 为什么接上行业主线会显著增强项目说服力

## 2. [OpenAI Agents SDK](https://platform.openai.com/docs/guides/agents-sdk/)
- 为什么这个来源重要：它能帮助你用更主流、更清晰的技术语言描述自己的 agent 实践。对转型来说，很多问题不在于你没做，而在于你不会把自己做的东西翻译成别人熟悉的术语体系。
- 这个来源原本在解决什么问题：
  - 如何描述 agent runtime
  - 如何组织 tools、handoff、trace
  - 如何用统一术语表达 agent 系统的关键构件
- 你应该从原文里抓到的核心原理：
  - 术语不是为了装专业，而是为了让你的经验能被快速对齐。
  - “我做了一个支持工具调用、可追踪、可治理的 runtime”会比“我写了很多 prompt 和接口”更准确。
  - 语言一旦更靠近行业主流，项目表达就更容易获得信任。
- 本教程吸收成了哪些内容：
  - `infra 版项目讲法`
  - 面试问答包里的 runtime、tool、trace 相关表述
  - 为什么课程要让你到最后还能回指代码、回指运行时结构
- 读原文时重点看什么：
  - runtime
  - tool
  - handoff
  - trace
  - 这些术语如何构成一套完整叙事
- 读完后你应该能回答：
  - 为什么“术语翻译能力”会影响面试表现
  - 为什么同样项目，用更准确的 runtime 语言讲出来会更像工程实践

## 3. [LangGraph overview](https://docs.langchain.com/oss/python/langgraph/overview)
- 为什么这个来源重要：它帮助你在讲项目时解释为什么需要 workflow orchestration。很多面试官不会满足于你说“我用了某个框架”，他们更想听的是：为什么这里需要显式流程、为什么不能全交给自由 agent。
- 这个来源原本在解决什么问题：
  - 如何把多步流程做成显式图结构
  - 如何把状态和分支变成可见对象
  - 如何让复杂控制流变得可读、可控
- 你应该从原文里抓到的核心原理：
  - 选型表达的关键，是把痛点和结构对应起来。
  - 当你说“这里使用 workflow orchestration”，你必须能继续解释：它解决了哪些自由 agent 难以治理的问题。
  - 只有能讲清为什么，项目才会从“做过”变成“理解过”。
- 本教程吸收成了哪些内容：
  - `为什么这里用 workflow 而不是 agent`
  - 第 06 模块里技术面试版项目讲法
  - 失败案例讲述时对流程节点和状态变化的解释
- 读原文时重点看什么：
  - state
  - graph
  - orchestration
  - 为什么显式流程有助于控制复杂度
- 读完后你应该能回答：
  - 为什么 workflow 不是“更复杂”，而是“更可控”
  - 为什么技术面试里，结构理由往往比框架名更重要

## 这一页真正想帮你建立什么
- 转型表达不是凭感觉讲故事，而是把你的项目重新映射到行业语境
- 好的项目讲法，会同时包含：技术结构、失败治理、行业上下文
- 你回看这些来源时，应该不断问：这能帮我把自己的项目讲得更像真实工程吗
