## 1. 这个模块真正要解决什么问题
绝大多数人学习 agent infra 的失败，不是败在模型或框架本身，而是败在一开始没有建立“系统观”。如果你把 agent infra 理解成“写 prompt + 调 API”，那么后面看到 tool calling、RAG、workflow、trace、eval 时，就会像在看互不相干的技术点。这个模块的任务是先把地图建起来。

所谓 agent infra，本质上是在回答 5 个问题：
1. 模型怎么接入。
2. 外部能力怎么调用。
3. 知识和上下文怎么进入系统。
4. 过程怎么被控制、观测和评估。
5. 这一切如何变成可维护的工程结构。

## 2. 什么是 agent infra
`agent infra` 不是单个框架，也不是单个 API，而是一套围绕 agent 系统运行的工程基础设施。它至少包含：
- model interface：模型接口层
- tool interface：外部能力接入层
- context layer：记忆、检索、输入上下文
- runtime：状态流转和执行编排
- observability：日志、trace、元数据
- eval：质量、稳定性、成本、延迟的验证方式

你可以把它类比成传统后端里的“应用框架 + 中间件 + 监控 + 测试 + 配置治理”，只是服务对象从 HTTP 请求变成了“模型参与的复杂执行流程”。

## 3. 为什么课程从配置、日志、状态检查开始
因为 agent 系统的复杂度增长非常快。你后面会遇到模型名、温度、工具开关、检索参数、超时、回退策略、trace 开关、评测数据集等大量变量。如果第一周不建立统一的配置和记录方式，后面就一定会在“为什么今天和昨天表现不一样”这种问题上浪费时间。

## 4. 这一阶段要建立的三个基本原则
### 原则一：先有系统边界，再有能力堆叠
不要一开始就做 multi-agent、数据库、前端页面。先把运行边界、配置入口和验证方式固定下来。

### 原则二：所有学习都要留下证据
这里的证据包括：实验记录、失败案例、trace、代码、测试、总结。只靠“我好像懂了”是无法进入 agent infra 工程状态的。

### 原则三：主项目必须持续演进
课程不是 6 个互相独立的小练习，而是一条逐步长大的主项目主线。只有这样，你才能真的理解后面为什么要引入 RAG、workflow 和 eval。
