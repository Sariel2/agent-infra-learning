# 00 总览与准备 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一篇不是正式进入某个具体技术点，而是先帮你建立一张总图。很多工程师第一次学 agent infra 时，最自然的动作是直接写一个 prompt，然后调一次模型接口看看结果对不对。这一步当然有价值，但如果你的目标是做一个长期可维护、可迭代、可调试的 agent 系统，那么只会 prompt 远远不够。

这个模块真正要建立的是一套总框架：以后你看到的每个能力点，都要知道它在系统里属于哪一层，解决什么问题，和其他层是什么关系。

## 1. 为什么学习 agent infra 要先学“系统图”
agent infra 的真正难点，不在“模型会不会回答”，而在于：
- 模型怎样与外部工具协同
- 知识怎样进入上下文
- 系统怎样决定下一步动作
- 失败时怎样定位问题
- 改动以后怎样判断系统是否真的变好

如果一开始没有系统图，你后面学到的每个新概念都会像散点：
- 今天学 tool
- 明天学 RAG
- 后天学 eval
- 再后天学 multi-agent

最后词汇越来越多，但脑子里没有主线。系统图的作用，就是给这些能力一个统一坐标系。

## 2. 什么是 agent infra
你可以把 agent infra 理解成“围绕 agent 系统运行而建立的一整套工程基础设施”。它至少包含下面几层：

### 模型接口层
负责和模型交互，定义请求、响应、结构化输出和模型接入方式。

### 外部能力层
负责把工具、内部 API、数据库、搜索、文件系统等能力接进系统。

### 上下文层
负责管理输入上下文、知识库、检索结果、记忆和约束信息。

### 运行时层
负责状态流转、步骤编排、动作执行和多轮循环。

### 观测层
负责日志、trace、元数据、调试信息和故障定位。

### 评测层
负责验证系统是否更准确、更稳定、更可控、更便宜。

这些层和传统后端工程很像：
- 模型接口层有点像外部依赖适配层
- 运行时层有点像业务编排层
- 观测层和评测层类似监控、测试和质量保障体系

区别在于：这里服务的不是简单 CRUD，而是带有模型参与和非确定性的复杂流程。

## 3. 为什么课程从环境、配置、记录机制开始
很多人会轻视这些内容，觉得“先配环境、写日志、建模板”只是准备动作，不是核心学习。但对 agent infra 来说，它们其实是后面所有复杂能力的基础。

原因很简单：agent 系统的可变因素特别多。后面你会碰到：
- 模型名
- 温度和超时
- retry 次数
- tool 开关
- 检索参数
- chunk 策略
- trace 开关
- eval 数据集版本

如果这些一开始没有被记录和组织起来，后面你会频繁遇到一种非常痛苦的状态：
- 系统表现变了
- 但你不知道到底是哪一层变了
- 也不知道这次好转是偶然，还是某个参数真的更合理

所以课程把配置、记录、模板放在最前面，不是形式主义，而是在帮你提前建立“证据体系”。

## 4. “会跑一个 demo”和“进入 agent infra 学习状态”的区别
这两者差别很大。

### 会跑一个 demo
通常意味着：
- 能拿到一个看起来对的输出
- 能让接口返回结果
- 能向别人演示“这个功能有点意思”

### 进入 agent infra 学习状态
意味着：
- 你知道系统有哪些层
- 你知道一条输出是怎么来的
- 你知道失败会落在哪一层
- 你知道应该保留哪些证据帮助后续定位
- 你知道如何让下一个模块建立在当前模块之上，而不是每周重新起一个新 demo

课程真正要带你进入的是第二种状态。

## 5. 为什么主项目必须持续演进
如果每周都换一个全新 demo，看起来会学很多，但实际上你是在反复做“重新起步”。这样的问题是：
- 你感受不到复杂度怎样逐层叠加
- 你也不知道某个能力为什么必须在某个阶段引入
- 失败案例会散成零碎体验，难以形成系统认知

持续演进同一个主项目，带来的价值包括：
- 你能看到系统从单次调用变成 tool-aware service
- 再从 tool-aware service 变成 knowledge-aware service
- 再从 knowledge-aware service 变成带 workflow、trace、eval 的 runtime
- 最后还能被组织成可展示、可面试的完整资产

这条主线，会让学习过程更像真正的软件演化，而不是一堆孤立教程拼盘。

## 6. 为什么课程一定要区分“能力点”和“治理点”
这是整套教程背后的重要设计原则。

### 能力点
指系统“能做什么”，例如：
- 结构化输出
- tool calling
- retrieval
- memory

### 治理点
指系统“怎样被稳定维护”，例如：
- 配置
- trace
- eval
- 测试
- 失败记录

很多教程只讲能力点，最后导致学习者会的东西很多，但系统一复杂就开始失控。真正的工程能力，恰恰来自能力点和治理点一起增长。

## 7. 为什么这门课不从 multi-agent 开始
从吸引力上看，multi-agent 很容易被当成“最厉害的部分”。但从学习顺序上，它绝对不适合放在最前面。

因为 multi-agent 会把这些问题全部放大：
- 状态管理
- 上下文同步
- trace 复杂度
- 失败归因
- 评测难度

如果单 agent、tool、retrieval、workflow、eval 这些基础能力都还没有稳定，直接上 multi-agent，通常只会把学习过程变成“热闹但混乱”。

## 8. 这一模块真正要建立的三个思维习惯
### 习惯一：任何能力都要能放回系统层次里
以后看到 tool、memory、RAG、workflow、trace、eval 时，不要把它们当成独立技巧，而要问：
- 它属于哪一层
- 它依赖哪一层
- 它会影响哪一层

### 习惯二：任何实验都应该留下证据
证据包括：
- 运行结果
- 失败样本
- 配置快照
- 测试
- trace
- 结论记录

没有证据，后面你只会留下模糊印象。

### 习惯三：任何“感觉有效”的优化都不值得立刻相信
后面你会经常产生一种感觉：
- “这次好像更好了”
- “这个 prompt 看起来更稳”
- “这个 chunking 感觉更准”

如果没有后续的 trace、对比和 eval，这些感觉通常并不可靠。现在就建立证据意识，后面会省下很多时间。

## 9. 这一模块和后续模块的关系
这一模块不是独立导论，而是后面所有模块的框架底板：
- 第 1 模块会往模型接口层和服务分层里走
- 第 2 模块会进入 tool calling 和 observation
- 第 3 模块会进入 knowledge、retrieval 和 memory
- 第 4 模块会补齐 workflow、trace、eval
- 第 5 模块会回头抽象成 pattern
- 第 6 模块会把前面所有东西变成可展示资产

所以这个模块虽然不炫，但它决定你后面是“看懂一门课”，还是“学到一堆词”。

## 10. 学完本篇的完成标准
不是“我大概知道 agent infra 有哪些词”，而是：
- 你能画出一张系统层次图
- 你能解释课程为什么按这个顺序展开
- 你能说明为什么配置、日志、记录机制不是附属工作
- 你知道主项目为什么必须持续演进
- 你已经开始用“能力点 + 治理点”的视角看系统

只要这几点建立起来，后面整套课就会顺得多，因为你已经拥有了一张不会轻易丢的总地图。
