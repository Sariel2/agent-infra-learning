# 01 Python 与 Agent 基础 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一模块的重点不是“学会 Python 语法”，而是建立一个适合 agent infra 开发的 Python 工程心智。也就是说，你不是来成为 Python 语言专家，而是来回答这个问题：

一个最小的、可维护的、能逐步演化成 agent runtime 的 Python 服务，应该长什么样？

这个问题背后有三个关键词：
- 结构化
- 可测试
- 可替换

## 1. 为什么 Python 基础模块不是语法速查表
如果你已经是后端工程师，Python 里真正会卡住你的，通常不是 `for` 循环或列表推导，而是这些工程层面的问题：
- 怎样组织项目结构
- 怎样表达稳定的数据契约
- 怎样把外部模型接口隔离起来
- 怎样让路由层不污染业务层
- 怎样给后面 tools、RAG、workflow 预留演化空间

所以这模块真正要建立的，是“Python 如何承载 agent infra”的直觉，而不是零散语法点。

## 2. 为什么结构化输出是第一块地基
很多人第一次接模型，会让模型直接输出一段自由文本。自由文本对“让人看”很友好，对“让系统消费”很不友好。

系统需要的是：
- 字段稳定
- 类型稳定
- 语义边界清楚
- 下游逻辑能依赖它继续执行

这就是结构化输出的核心价值。它不是“格式更漂亮”，而是“系统边界更稳定”。

### 自由文本的问题是什么
如果模型直接返回一段自然语言，下游会遇到几个问题：
- 不知道哪些信息一定存在
- 不知道字段顺序是否稳定
- 不知道能否可靠提取置信度、分类结果、工具参数等结构
- 测试只能做非常脆弱的字符串匹配

### 结构化输出在工程里的意义
当你定义了 `GenerateResponse`、`StructuredAnswer` 这类对象时，你实际上是在做两件事：
1. 给模型输出建立契约
2. 给系统其他模块建立依赖边界

后面的 tool result、RAG answer、eval result 都会延续这个思路。

## 3. Provider abstraction 的本质
很多新手以为 provider 只是“把 API 调用包一层”，但它真正的价值在于隔离变化。

在模型相关系统里，最容易变化的恰恰是接入层：
- 模型名称会变
- 供应商会变
- 请求参数会变
- 错误处理会变
- 超时/重试策略会变
- tracing 和 metadata 接入方式会变

如果这些变化直接进入 route 或业务逻辑，系统会很快失控。

所以 provider 的本质是：
- 对外提供稳定能力
- 对内吸收模型接入变化

它的价值不在“包一层”，而在“挡住变化”。

## 4. Route、Service、Provider 三层为什么要分开
### Route 层
负责 HTTP 协议边界：
- 接收请求
- 解析参数
- 返回响应

### Service 层
负责业务编排：
- 决定调用哪个 provider
- 决定是否做附加转换
- 决定是否触发额外逻辑

### Provider 层
负责模型接入与适配：
- 封装模型 API
- 统一请求/响应
- 隔离第三方变化

这种分层的价值在于：
- route 保持简单，便于测试 API 协议
- service 可以独立于 HTTP 存在，便于复用和重构
- provider 可以被 fake、mock 或真实实现替换

后面 tools、RAG、workflow 之所以能逐步长进去，很大程度上就是因为这里先把边界立住了。

## 5. 为什么先 fake，再 real
这是这一模块最重要的工程习惯之一。

### 一开始直接接真实模型的问题
如果你一开始就接真实模型，任何问题都有可能来自：
- 模型接口没接对
- schema 设计有问题
- route 和业务逻辑耦合
- 外部依赖不稳定
- 测试环境不可控

这会让你很难判断：当前问题到底是结构错了，还是外部系统本身在波动。

### fake provider 的价值
先用 fake provider，可以先验证：
- 输入输出契约是不是合理
- service 边界是不是清楚
- route 是否承担了过多逻辑
- 测试是否能独立运行

它的本质，是先把结构问题和外部依赖问题分开。

## 6. 为什么 Pydantic 和类型约束在这里特别重要
agent infra 不是“脚本随便跑一下”的系统。后面你会越来越依赖：
- request schema
- response schema
- tool schema
- eval case schema
- trace metadata schema

如果没有稳定的数据模型，整个系统会逐渐变成“到处传 dict，靠约定做人肉理解”。Pydantic 在这里最重要的价值，不是少写样板代码，而是让数据契约变成明确、可验证、可测试的对象。

## 7. 为什么 service 层是未来演进的关键位置
service 层的价值在于，它最适合承接“业务决策”。

后面很多演进都会发生在这里：
- 从单一模型响应到 tool dispatch
- 从单次调用到多步 workflow
- 从直接回答到 RAG + memory + citation

如果一开始没有 service 边界，这些演进会全部挤进 route 或 provider，系统会迅速变形。

## 8. 为什么 Python 是这套课程的主栈
这里不是在否定 Java，而是在承认当前 agent infra 生态的现实：
- AI 相关库首发更集中在 Python
- 实验、脚本、服务、评测经常共享同一语言生态
- 用 Python 建立原型和小型运行时成本更低

所以这套课程用 Python，不是为了跟风，而是为了降低进入 agent infra 核心问题的阻力。

## 9. 常见误区
### 误区一：会调 OpenAI API 就等于掌握了这一模块
不是。调 API 只是起点，真正重要的是能不能把它变成可维护的服务结构。

### 误区二：provider 只是形式上的抽象
如果 service 仍然依赖具体实现，或者 route 直接透传模型细节，那这个抽象就没有真正发挥作用。

### 误区三：结构化输出只是让 JSON 更漂亮
结构化输出真正解决的是契约稳定性，不是美观问题。

### 误区四：fake provider 是临时凑合
恰恰相反，它是隔离外部依赖、验证结构正确性的关键训练手段。

## 10. 学完本篇的完成标准
你应该能讲清：
- 为什么结构化输出是第一块地基
- 为什么 route、service、provider 要分层
- 为什么 provider 的本质是隔离变化
- 为什么先 fake 再 real 是合理的工程策略
- 为什么后面的 tools、RAG、workflow 都会依赖这里的骨架

只要这些问题能回答清楚，这一模块就不只是“学了点 Python”，而是真正打下了 agent infra 服务骨架。
