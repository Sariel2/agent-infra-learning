# 01 Python 与 Agent 基础 细节深挖

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：01-concepts-and-principles.md](./01-concepts-and-principles.md) | [下一篇：03-code-reading-guide.md](./03-code-reading-guide.md)

这一篇从运行链路和工程演化角度，把上一节的原则再往下拆。核心目标是：让你知道最小 Python agent service 在代码里到底应该怎么流动，以及为什么这个流动方式会成为后面所有模块的共同底板。

## 1. 从请求到响应的完整数据流
这一模块的最小链路是：
1. HTTP 请求进入 route
2. route 解析成请求对象
3. service 调用 provider
4. provider 返回结构化响应对象
5. route 再把它返回给调用方

这个链路之所以重要，是因为它几乎会出现在后面所有模块里。到了 tools、RAG、workflow、eval，你仍然是在这个基础链路上不断增加新层次，而不是推翻重来。

所以你现在学的不是“一个临时小例子”，而是一条以后会反复复用的最小主干。

## 2. 为什么 route 不应该直接调用模型
当 route 直接调用模型时，看起来很快，但会立刻带来几个工程后果：
- 无法在不启动 HTTP 的情况下复用业务逻辑
- 无法轻松替换 fake provider 和 real provider
- 测试需要穿透太多层
- 模型接入变化会直接污染 API 层

这其实就是边界错位。route 最适合负责“协议接入”，不适合负责“模型执行业务”。

一旦 route 层开始承担太多事，后面每加一个功能点都会变得更难维护：
- 想加 tool dispatch，会塞进 route
- 想加 RAG，会塞进 route
- 想加 trace 或 metadata，也会塞进 route

所以现在把 route 控制住，后面会轻松很多。

## 3. 结构化输出如何影响测试策略
有了结构化输出之后，测试方式会明显变化：
- 你不再主要断言一整段字符串是否完全一致
- 你会断言字段是否存在、类型是否正确、值是否落在合理范围
- 你可以稳定地验证业务行为，而不是文案表面

这意味着测试不再那么脆弱，重构空间也更大。

例如，后面你换模型、换文案、换提示词时，只要结构契约没变，很多测试仍然可以成立。这就是为什么结构化输出不只是运行时便利，也是测试稳定性的前提。

## 4. provider abstraction 如何帮助后续模块
这一层不是这模块里的临时技巧，而是后续模块会持续复用的一条主线。

### 在 tools 模块
provider 可以承接“模型如何生成动作”。

### 在 RAG 模块
provider 可以承接“模型如何综合检索上下文并生成响应”。

### 在 workflow 模块
provider 可以承接“某个节点里的模型执行逻辑”。

也就是说，provider abstraction 让“模型能力”成为一个可被组合的部件，而不是一段散落在各处的 API 调用。

## 5. 为什么 service 层是未来演进的关键位置
service 层的价值在于，它最适合承接“业务决策”。后面很多演进都会发生在这里：
- 从单一模型响应到 tool dispatch
- 从单次调用到多步 workflow
- 从直接回答到 RAG + memory + citation

service 层像一个“缓冲层”：
- 上面是协议边界
- 下面是具体外部能力

正因为有这层缓冲，后面你才能在不重写整个接口层的情况下，不断往系统里加新能力。

## 6. fake provider 和 real provider 的切换到底在验证什么
很多人以为两者切换只是为了方便开发。实际上，它在验证一个更关键的问题：

“你的系统设计有没有把外部依赖从内部结构里分离出来？”

如果能平滑切换 fake 和 real provider，通常说明：
- route 不依赖底层供应商细节
- service 更依赖能力契约，而不是具体实现
- 测试可以优先验证结构，再验证真实集成

如果切换很痛苦，就说明抽象其实没有立住。

## 7. 这个模块最常见的反例
### 反例一：所有逻辑都在 route
结果是 HTTP 层和业务层完全耦合，后面任何扩展都痛苦。

### 反例二：provider 名义上存在，但 service 仍然依赖具体实现
这意味着 provider abstraction 只是“换了个文件名”，没有真正实现隔离变化。

### 反例三：结构化输出只是表面 JSON
如果系统仍然把它当字符串拼接使用，那么 schema 的价值没有真正落地。

### 反例四：一开始就只用真实模型做测试
这样会让结构问题和外部波动问题混在一起，导致定位效率很低。

## 8. 这一模块和后面模块的真正连接点
如果你后面学 tools、RAG、workflow 时感到顺手，往往不是因为后面模块写得简单，而是因为这里已经把最小服务主干搭好了。

后面的演进可以被理解成：
- 第 2 模块：在 service/provider 之间引入工具动作和 observation
- 第 3 模块：在 provider/service 里引入 retrieval、memory、citation
- 第 4 模块：在 service 外再引入显式 workflow、trace、eval

如果这里的分层没有立住，后面的每一步都会显得更乱。

## 9. 怎么判断这一模块骨架搭得是否健康
你可以用几个问题来自检：
- route 是否只负责协议边界
- service 是否承接业务决策
- provider 是否吸收模型接入变化
- request/response 是否有明确 schema
- fake provider 是否能支撑大部分结构测试

只要这些问题大体都能答“是”，说明骨架已经比较健康。

## 10. 学完本篇的完成标准
不是“我能调通一个接口”，而是：
- 你能解释每一层为什么存在
- 你知道未来 tools 和 RAG 会接在哪一层
- 你知道为什么 fake provider 是合理起点
- 你已经建立了可测试、可替换、可扩展的最小骨架
- 你能看懂后面模块为什么能建立在这里，而不是重新起盘

一旦达到这个标准，这一模块就真正完成了。因为它提供的不是零散 Python 技巧，而是一条后面所有模块都会依赖的服务主干。
