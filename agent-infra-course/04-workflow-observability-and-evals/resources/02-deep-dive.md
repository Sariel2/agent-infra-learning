# 04 Workflow、Observability 与 Evals 细节深挖

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：01-concepts-and-principles.md](./01-concepts-and-principles.md) | [下一篇：03-code-reading-guide.md](./03-code-reading-guide.md)

这一篇继续往下拆实现细节。核心问题是：workflow、trace、eval 在代码和运行时里到底应该长什么样，以及它们为什么必须互相配合。

## 1. workflow 和 agent 的边界，不是二选一，而是分工
判断二者边界时，可以先问 4 个问题：
- 步骤顺序是否大体固定
- 分支是否可以被预先建模
- 失败代价是否较高
- 是否存在明确的人类审核点

如果这些问题的答案大多是“是”，workflow 的价值通常更高。因为它能把状态转换、分支和中断点显式化。

但如果中间某个节点存在明显开放决策，例如：
- 要不要继续搜资料
- 该调用哪个工具
- 该追问用户还是直接返回

那就可以把这一段开放性包进一个 agent 节点里。于是系统变成：
- 外层 workflow 控制大框架
- 内层 agent 解决局部开放决策

这就是课程反复强调“workflow 包住 agent”的原因。

## 2. trace 中最值得记录的元数据
一个可用 trace 至少要让你事后能复盘“这次运行发生了什么”。因此以下元数据通常非常关键：
- request id：定位单次请求
- run id：区分同一请求下的多次实验
- node name：知道当前在哪一步
- elapsed time：看出瓶颈节点
- tool name / retrieval info：知道具体调用了什么
- status：区分成功、失败、回退
- experiment tag：知道这次属于哪个版本或配置

没有这些字段时，你可能只能知道“某次失败了”，却不知道：
- 是哪个节点失败
- 是哪版逻辑失败
- 是工具慢还是模型慢
- 是局部异常还是全局路径设计有问题

## 3. eval dataset 应该长什么样
一个最小 eval case 至少应该包含：
- 输入
- 期望结果或期望行为
- 评估规则
- 可选标签
- 可选备注

这里最容易被忽略的是“期望行为”不一定总是“精确答案”。在 agent 系统里，你有时更关心这些事情：
- 是否走了正确节点
- 是否用了正确工具
- 是否附上 citation
- 是否在失败时收束到保守输出

这说明 eval 不是只有结果对错一种形式。对 agent infra 来说，过程正确性和结构正确性同样重要。

## 4. 为什么“感觉系统变好了”不可靠
agent 系统波动性本来就大。你今天试了 5 个例子看起来都变好了，并不代表：
- 其他路径没变差
- 延迟没增加
- 成本没变高
- 某些边角 case 没退化

所以 eval 的意义不是追求绝对真理，而是建立一个更可靠的比较机制。哪怕 dataset 很小，只要它固定、可重复，也比“随机手测几个问题”要强得多。

## 5. workflow 图、trace、eval 为什么会经常脱节
这在真实系统里非常常见：
- 文档里画了一套流程图
- 代码里跑的是另一套分支
- trace 里节点名又和代码不一致
- eval 最后评的还是一堆手工 case

一旦三者脱节，系统治理就会非常痛苦。因为你根本无法回答：
- 这次失败，是设计错了，还是实现偏了
- 这次优化，是路径变好了，还是只是评分口径变了

课程里之所以把 `agent-runtime` 做成统一入口，就是为了让：
- workflow 结构
- trace 节点
- eval case

至少在一个教学级系统里保持同一语言体系。

## 6. 常见反例
### 反例一：有 workflow 图，但代码没边界
结果流程图只是展示物，不是真正治理工具。代码里的逻辑仍然散在多个文件和多个 if/else 里，改动后也没人知道图还对不对。

### 反例二：有 trace，但信息不够
你能看到“失败了”，但仍然不知道：
- 失败在哪层
- 为什么失败
- 哪个版本开始失败

这种 trace 只能算“增强日志”，还称不上真正可观测。

### 反例三：有 eval，但没有固定 dataset
每次测试样本都不一样，最后只能得到一种错觉：好像变好了，但没有稳定证据。

### 反例四：只测最终答案，不测过程
结果某次系统碰巧答对了，但其实用错了工具、走错了路径、引用也不一致。只看最终输出会掩盖很多真实问题。

## 7. 为什么 agent-runtime 是这一模块的核心收束
`agent-runtime` 把前面学到的能力第一次真正统一成“运行时系统”：
- workflow 不再只是图，而是执行骨架
- trace 不再只是概念，而是运行证据
- eval 不再只是想法，而是回归机制

这说明你的关注点开始从“我会做某个能力”转向“我会组织并维护这些能力”。

## 8. 从排障角度看，这一模块到底教了什么
如果一个系统输出变差了，这一模块训练你按这样的顺序思考：
1. workflow 有没有走错分支
2. trace 里哪一步开始偏离预期
3. 这个偏离是数据问题、工具问题，还是节点逻辑问题
4. eval dataset 是否覆盖了这个场景
5. 下次如何把这个问题沉淀成可重复检测的 case

这套思路，就是 agent 系统真正的工程化思路。

## 9. 本模块和第 5 模块的连接点
这一模块解决的是“系统如何被运行和治理”，下一模块会进一步问：
- 为什么这里适合 workflow
- 为什么那里应该保留 agent
- 为什么某些地方不值得上 multi-agent

也就是说，第 5 模块的 pattern 判断，其实建立在这里的运行时感知之上。如果你没有 workflow、trace、eval 的直觉，就很难做出稳的架构判断。

## 10. 学完本篇的完成标准
你应该能拿一个失败样例，明确说出：
- 该先看 workflow 哪个节点
- trace 里该找什么信息
- eval 该补什么 case
- 下次怎样防止同类问题再次悄悄出现

只要你能这样思考，说明你已经不再是“功能视角”看 agent，而是开始从“运行时治理视角”看系统。
