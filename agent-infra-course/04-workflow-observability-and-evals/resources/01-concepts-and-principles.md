# 04 Workflow、Observability 与 Evals 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

## 1. 为什么 workflow 必须显式存在
当系统开始同时拥有工具、检索、总结、人工审核、回退策略时，如果没有显式 workflow，你最终只会得到一团“能跑但不可解释”的流程代码。workflow 的价值不是更炫，而是把这些步骤显式建模出来。

适合 workflow 的信号包括：
- 步骤顺序基本固定
- 某些节点失败代价高
- 某些节点需要人工审核
- 你希望稳定比较流程优化前后结果

## 2. trace 的本质
trace 不是更详细的日志，而是“把一次运行拆成可追踪的结构化步骤”。对于 agent 系统来说，这尤其重要，因为很多错误都发生在“中间过程”，而不是最终回答本身。

一个最小 trace 至少要能回答：
- 走了哪些节点
- 每个节点输入是什么
- 每个节点输出是什么
- 结果是成功还是失败
- 关键元数据是什么

## 3. eval 的本质
eval 的目标不是做一个漂亮分数，而是建立“修改后能否更好”的比较机制。没有固定数据集和固定指标，你每次都只能靠主观感觉比较，这在 agent infra 里很快会失效。

## 4. 为什么 workflow、trace、eval 是一组连在一起的能力
- workflow 告诉你系统理论上应该怎么走
- trace 告诉你系统实际上怎么走
- eval 告诉你这样走出来的结果是不是更好

这三者组合起来，系统才真正可迭代。
