# 04 Workflow、Observability 与 Evals 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一篇是整条学习主线里非常关键的转折点。前面几个模块你已经分别接触了模型调用、tool calling、retrieval、memory，但到这里开始要回答更像生产问题的问题：这些能力如何被组织成稳定系统，如何被观测，如何被比较，如何在多次迭代后还能维持可治理性。

## 1. 为什么 workflow 必须显式存在
当系统只有一次模型调用时，流程几乎可以隐形存在。但当系统开始拥有：
- 分类
- tool dispatch
- retrieval
- 人工审核
- 总结
- fallback

如果这些步骤没有显式流程结构，最后就会退化成一团“能跑但没人说得清”的流程代码。

workflow 的价值不在于“更炫”，而在于把下面这些东西显式建模出来：
- 节点
- 状态
- 分支
- 重试点
- 中断点
- 人工介入点

一旦这些边界被显式化，系统才可能真正进入“可维护”状态。

## 2. workflow 和 agent 不是对立关系
这是最常见的误解之一。workflow 并不等于“没有模型参与”，agent 也不等于“所有步骤都自由”。

更准确的理解是：
- workflow 负责固定结构
- agent 负责局部开放性决策

很多稳定系统的真实形态，其实是“workflow 包住 agent”。例如：
- 入口先走固定分类流程
- 中间某个节点让 agent 自主选择工具
- 输出前进入固定审核或评估节点

这种组合式设计，往往比“要么全 workflow，要么全 agent”更接近真实工程。

## 3. trace 的真正含义
trace 不是更详细的日志。日志通常是面向人肉查看的文字记录，而 trace 是把一次运行拆成多个可追踪的结构化步骤。

它能回答的问题包括：
- 这次请求经过了哪些节点
- 每个节点的输入输出是什么
- 哪一步耗时最多
- 哪一步失败了
- 是否走了 fallback
- 工具调用或检索发生在哪个阶段

如果没有 trace，你最后看到的只有“结果”，很难看到“过程”。而 agent 系统最大的复杂度，恰恰往往藏在过程里。

## 4. eval 的真正含义
eval 不是为了追求一个漂亮分数，更不是为了给系统贴一个“准确率 xx%”的标签。它真正要回答的是：

“我改完系统之后，它是真的更好，还是只是碰巧在几个例子上更像对了？”

一个最小 eval 至少要有：
- 固定输入
- 期望结果或期望行为
- 稳定判断规则

如果没有固定样本和稳定规则，每次迭代都只能靠感觉比较。感觉可以帮助你发现问题，但不能替代系统性比较。

## 5. 为什么 workflow、trace、eval 必须一起学
这三者其实在回答同一个系统的三个层面：
- workflow：系统理论上应该怎么走
- trace：系统实际上是怎么走的
- eval：这样走出来的结果是不是更好

只学其中一个，都会让理解不完整：
- 只有 workflow，没有 trace，你只知道理想结构，不知道真实运行
- 只有 trace，没有 eval，你知道发生了什么，但不知道结果是否真的变好
- 只有 eval，没有 workflow，你可能知道分数变了，却不知道该回哪层修系统

所以这一模块不是同时堆三个新词，而是在建立系统治理闭环。

## 6. observability 为什么比“加日志”复杂得多
可观测性关注的不是“信息多不多”，而是“你能否从现象稳定回溯到原因”。

在 agent 系统里，你通常至少要观测这些维度：
- 路径：走了哪些节点
- 时间：哪步慢
- 成本：哪步贵
- 工具：哪类调用容易失败
- 检索：命中了什么
- 实验标签：这次运行属于哪个版本或配置

如果信息只是散落的日志文本，后面很难稳定做比较、聚合和回归分析。

## 7. 为什么这是从 demo 走向 production-like 的关键模块
前面几个模块更多是在建立能力点：
- 会调模型
- 会接工具
- 会做检索
- 会维护短期记忆

到了这里，重点开始变化。你不再只是问“这个能力能不能工作”，而是开始问：
- 这个能力在系统里怎么编排
- 它失败时如何定位
- 它修改后如何验证
- 它上线后如何持续观察

这就是从 demo 走向 production-like system 的关键转折。

## 8. 常见误区
### 误区一：有流程图就等于有 workflow
不是。流程图只是说明文档，workflow 要真正进入执行结构、状态流和错误处理。

### 误区二：有 trace 就等于可观测
也不是。如果 trace 里没有关键元数据，没有稳定节点边界，还是很难定位问题。

### 误区三：有 eval 就说明系统被量化了
如果 dataset 不固定、规则不稳定，那这个 eval 只是一次性试验，不是治理工具。

### 误区四：只有大系统才需要 observability
恰恰相反。很多问题在系统还小的时候更容易建立良好结构，否则后面只会更难补。

## 9. 这一模块和前面模块的连接关系
这一模块其实是在给前面学过的能力装上“运行时骨架”：
- 第 2 模块的 tool call，会变成 workflow 中的某些节点
- 第 3 模块的 retrieval 和 memory，会进入 trace 和 eval 观察面
- 后续第 5 模块的 pattern，对这里的 workflow/agent 边界判断会进一步抽象

所以它不是独立新知识，而是把前面学过的零散能力系统化。

## 10. 学完本篇的完成标准
你应该能讲清：
- 为什么流程必须显式化
- 为什么 workflow 和 agent 是组合关系，而不是对立关系
- 为什么 trace 不等于日志
- 为什么 eval 不等于拍脑袋试几个 case
- 为什么 production-like 的关键不是更多功能，而是更强治理

只要这些判断建立起来，你就会开始用“系统维护者”的视角看 agent，而不只是“功能实现者”。
