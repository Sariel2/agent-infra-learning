# 04 Workflow、Observability 与 Evals 常见错误

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：04-sources.md](./04-sources.md)

下面这些错误不是“粗心”，而是这个模块里最常见的思维偏差和工程偏差。每次遇到问题时，你都可以先来这里对照排查。

## 1. 把所有步骤都交给模型决定
- 常见表现：系统不稳定且难以比较。
- 修正方式：把固定节点抽成 workflow，把开放点留给 agent。

## 2. 只有日志没有 trace
- 常见表现：最终仍然不知道中间过程出了什么问题。
- 修正方式：为每个关键节点建立结构化 trace。

## 3. 没有固定 eval 数据集
- 常见表现：每次比较都靠感觉。
- 修正方式：先准备一小组代表性样本。

## 4. trace 信息太少
- 常见表现：看见“失败了”，但不知道失败在哪。
- 修正方式：至少记录节点、输入、输出、状态和耗时。

## 5. 指标太多但没主线
- 常见表现：收集了很多数字，却无法指导优化。
- 修正方式：先抓最能反映目标的 2-4 个指标。
