# 05 Agent 原理与架构 概念、细节与原理

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [下一篇：02-deep-dive.md](./02-deep-dive.md)

这一篇不是“模式名词表”，而是要把你前 4 个模块里已经做过的工程实践，抽象成一套稳定的判断框架。学完这一篇，你应该开始具备一种能力：面对一个新需求时，不是先想“我能不能上 multi-agent”，而是先判断“这个问题的控制流本质是什么，最小足够复杂度在哪里”。

## 1. 为什么模式学习必须放在工程实践之后
很多人一上来就学 ReAct、Plan-and-Execute、Supervisor、Router，结果最后只记住了几个标签。这种学习方式的问题在于：模式一旦脱离真实系统，就会变成空壳概念。

模式真正有价值的前提，是你已经经历过这些具体问题：
- tool schema 不清，导致模型动作不稳定
- retrieval 能跑，但 trace 不可读
- workflow 节点越来越多，开始怀疑是否该交给 agent
- agent 自由度很高，但失败后不知道问题出在决策、工具、状态还是 observation

只有经历过这些具体困境，你才会明白模式不是“高级写法”，而是把某种反复出现的问题结构化地解决掉。

所以这一模块的任务不是背术语，而是做两件事：
- 给过去做过的系统命名
- 给未来要做的系统提供选型依据

## 2. 什么是 agent pattern
这里说的 pattern，不是框架里的 class 模板，也不是某个固定目录结构。它更接近软件架构里的“可复用控制流模式”。

一个 pattern 至少回答 4 个问题：
- 决策在什么时候发生
- 状态如何在步骤之间传递
- 工具或子任务如何组织
- 失败如何传播、如何收敛

所以 pattern 的本质，不是“长什么样”，而是“系统如何流动”。如果一个方案名字不同，但控制流本质一致，那它们通常属于同一类模式变体。

## 3. 为什么控制流比组件名更重要
很多教程讲模式时，容易把注意力放在这些表面差异上：
- 是不是多轮
- 有没有 planner
- 有没有一个 supervisor
- 有没有多个 agent

这些特征本身都不够。真正关键的是控制流。

例如：
- 一个 Router 系统的本质是先分类，再走确定路径
- 一个 ReAct 系统的本质是推理和行动交替前进
- 一个 Plan-and-Execute 系统的本质是先显式分解，再逐步消费计划
- 一个 Supervisor 系统的本质是把协调权从执行节点提升到上层控制者

如果你只看组件名，很容易误判。比如“有 planner”不一定就是 Plan-and-Execute，因为有些系统只是把 planner 当成一个辅助建议器，真正执行仍然是自由 agent loop。

## 4. 常见模式分别在解决什么结构性问题
### ReAct
ReAct 解决的是“边探索边推进”的问题。任务本身不能一开始完全拆清楚，但又不能完全盲走，于是系统采用一种交替推进的方式：
- 先基于当前上下文做局部推理
- 选择一个动作
- 观察动作结果
- 再根据 observation 决定下一步

它最适合的不是“所有开放任务”，而是局部开放、但每一步都可以被 observation 收敛的任务。比如查订单、查文档、再决定是否创建工单。

### Router
Router 解决的是“入口混杂，但下游路径相对稳定”的问题。用户请求类型很多，但一旦判断出类型，后续处理链条通常比较固定。

经典场景包括：
- 客服问题先分成售前、售后、知识问答、工单请求
- RAG 请求先分成资料问答、流程问答、系统状态问答

Router 的价值不在于“智能”，而在于把复杂度前置到入口，让下游保持稳定。

### Plan-and-Execute
Plan-and-Execute 解决的是“任务链条较长，且中间步骤显式化会显著提升稳定性”的问题。它的核心不是 planner 本身，而是把“中间计划”提升成一个可观察、可修改、可验证的结构。

它通常比 ReAct 更适合这些场景：
- 子任务依赖关系比较明显
- 希望人工审核中间计划
- 希望失败后从某一步恢复，而不是整条链重来

### Supervisor
Supervisor 解决的是“多个执行单元之间需要协调，但不希望它们完全自由互相对话”的问题。它强调把协调逻辑集中到一个控制层，而不是让每个 agent 自己乱协商。

如果没有 supervisor，系统常见退化方式是：
- 每个 agent 都觉得自己有权扩展任务
- 状态边界混乱
- trace 很难看懂
- 错误归因越来越困难

## 5. workflow、agent、multi-agent 的边界到底是什么
这是整门课最值得反复练习的判断题。

### workflow
workflow 适用于：
- 步骤相对固定
- 分支条件可枚举
- 风险高，需要可预测性
- 人工审核点清晰

workflow 的优势是稳定、可测试、可观测。它的代价是灵活性不够。

### agent
agent 适用于：
- 某些关键步骤存在开放决策
- 工具选择有不确定性
- observation 会显著影响下一步
- 手写分支成本开始高于放权成本

agent 的优势是适应性。代价是系统行为更难预测，所以你必须用 schema、trace、eval 去约束它。

### multi-agent
multi-agent 只适用于更少数的场景：
- 任务天然可以拆成不同角色或能力域
- 单一上下文已经过大，拆分上下文确实能降复杂度
- 角色之间的收益大于协调成本
- 你有足够强的 tracing 和 evaluation 去治理多主体系统

很多系统并不需要 multi-agent。它不是“更高级”，只是“更贵的组织方式”。

## 6. 为什么 multi-agent 不是默认升级路线
很多新人会形成一个危险的错觉：单 agent 是初级版，multi-agent 是高级版。这个想法几乎总会把系统带偏。

multi-agent 带来的不只是更多角色，还会同步带来：
- 更多上下文边界
- 更多消息传递
- 更多状态同步点
- 更多失败来源
- 更长的 trace
- 更难归因的责任链

举个最常见的反例：
- 需求其实只是“先分类，再调用一个工具，再返回结果”
- 这本来是 Router + workflow 就能稳定解决的
- 结果为了“显得高级”，拆成 classifier agent、planner agent、executor agent
- 最后系统变成了 3 倍复杂度，收益接近 0

这就是为什么课程在这个阶段反复强调“复杂度预算”。系统不是能做多复杂就该做多复杂，而是应该只承担必要复杂度。

## 7. 设计模式真正要输出的不是名词，而是判断原则
学完 pattern 后，你真正应该获得的不是“我知道 8 种模式”，而是下面这些判断原则：
- 这个需求的核心复杂度是在入口分流、局部决策，还是长链规划
- 计划是否值得显式化
- observation 是否足够稳定，能支撑下一步决策
- 人工审核点应放在计划前、执行中，还是结果后
- 状态是否应该集中维护，还是按角色拆分
- 如果引入更多 agent，收益是否真的大于 trace 与调试成本

这些原则比任何框架名都重要，因为框架会变，控制流判断不会那么快过时。

## 8. 常见误区
### 误区一：看起来更智能的方案就更好
不是。越自由的系统越需要治理。没有 tracing 和 eval 的高自由度 agent，往往只是更难 debug。

### 误区二：只要有工具调用，就是 agent
不是。很多系统只是 workflow + tool execution。是否属于 agent，关键看决策是否开放、observation 是否影响后续。

### 误区三：有多个角色就叫 multi-agent
也不是。如果这些角色只是代码分层，或根本没有独立决策权，那它更像模块化 workflow，而不是真正的 multi-agent 协作。

### 误区四：模式是互斥的
很多真实系统是混合模式。例如入口用 Router，具体执行部分用 ReAct，中间穿插人工审核 workflow。模式更像积木，不是互斥阵营。

## 9. 这一模块和前面模块的连接关系
这一模块不是脱离前 4 个模块的新知识，而是对它们的抽象。

你可以这样回看前面的内容：
- 第 2 模块里的 support-agent，本质上接近单 agent / ReAct 风格
- 第 3 模块里的 knowledge-agent，更接近 retrieval + bounded state 的 workflow/agent 混合体
- 第 4 模块里的 agent-runtime，开始把状态机、trace、人工审核显式化，这是往 workflow orchestration 方向发展

所以你不是从零开始学模式，而是在给已经做过的东西建立语言。

## 10. 学完本篇的完成标准
如果你真的学会了，不是背出定义，而是能回答这些问题：
- 为什么这个需求更适合 Router，而不是自由 agent
- 为什么这里要先显式计划，再执行
- 为什么这里其实不该拆成 multi-agent
- 为什么某个系统看起来“高级”，其实只是复杂度失控

只要你能把这些判断讲清楚，你的 agent 架构认知就开始从“框架使用者”走向“系统设计者”。
