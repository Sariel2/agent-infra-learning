## 1. 为什么要在做完系统后再学模式
因为模式的价值不在于记住名字，而在于帮助你解释已经做过的事情，并在未来更快做判断。如果没有前面的实践，模式会变成空术语。

## 2. 常见模式分别在解决什么问题
### ReAct
适合“模型需要在推理与行动之间来回切换”的场景。它强调的是交替，而不是固定流程。

### Router
适合“先分流，再交给不同路径”的场景。核心问题是分类和路由，而不是多步推理。

### Plan-and-Execute
适合任务较长、需要先拆计划再执行的场景。它的关键是计划层和执行层分离。

### Supervisor
适合多个执行体需要被上层协调的场景，但复杂度明显更高。

## 3. workflow vs agent 的判断原则
这是本模块最重要的问题。你可以先看三件事：
- 步骤是否固定
- 决策是否开放
- 失败成本是否高

如果步骤固定、失败成本高，就优先 workflow。
如果需要开放性探索和工具选择，agent 才更有价值。

## 4. 为什么 multi-agent 不是默认更高级
多 agent 带来的不是自动收益，而是额外协调成本、上下文同步成本和调试成本。只有当单 agent 或 workflow 确实无法承载任务结构时，multi-agent 才值得考虑。
