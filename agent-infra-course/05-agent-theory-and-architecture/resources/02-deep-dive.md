# 05 Agent 原理与架构 细节深挖

[返回学习地图](../study-map.md) | [返回模块首页](../README.md) | [上一篇：01-concepts-and-principles.md](./01-concepts-and-principles.md) | [下一篇：03-code-reading-guide.md](./03-code-reading-guide.md)

这一篇要解决的不是“模式有什么定义”，而是“模式在实现里到底体现成什么”。如果上一节关注的是判断原则，这一节关注的就是控制流、状态、边界和失败传播。

## 1. pattern 的核心不是形式，而是控制流
很多人区分模式时，只看“是不是多轮”“是不是多个组件”“有没有 planner”。这些都太表面了。真正应该看的，是控制流：
- 谁拥有当前决策权
- 谁保存状态
- 哪个节点可以发起下一步
- 出错时系统在哪一层停止或回退

为什么这很重要？因为模式之间最本质的差异，就体现在控制流位置不同。

比如：
- ReAct 把决策权分散在每一轮 observation 之后
- Plan-and-Execute 把第一轮重要决策前置成显式计划
- Router 把主要决策集中在入口分类点
- Supervisor 把协调权集中到一个上层控制者

如果你看不到“决策权在哪里”，就很容易把不同模式混成“都差不多”。

## 2. 状态到底应该放在哪里
这是架构设计里最容易被忽略、但最容易引发混乱的部分。

### ReAct 风格
状态通常围绕当前会话或当前任务逐轮累积。重点是 observation 必须足够结构化，否则下一轮推理没有可靠输入。

### Router 风格
状态通常比较轻。核心状态是分类结果和路由依据，而不是长链上下文。

### Plan-and-Execute 风格
状态里会显式出现 plan、step index、step result、remaining work 等对象。也就是说，计划本身成为状态的一部分。

### Supervisor 风格
状态会分层：
- 顶层保存全局任务进度
- 子执行单元保存局部上下文
- 协调层负责决定什么时候合并、什么时候继续拆

一旦你把这些状态全揉在一起，系统就会开始出现两个问题：
- 没人知道哪一层该对错误负责
- 无法定位某次错误是局部失真，还是全局控制逻辑出错

## 3. 为什么课程里专门做 boundary checker
`architecture-lab` 里的 boundary checker，不是为了增加一个功能点，而是为了训练最关键的架构能力之一：复杂度控制。

工程里最贵的错误之一，不是“功能没做出来”，而是“做成了一个超出实际需要的复杂系统”。常见情形包括：
- 明明步骤固定，却用了自由 agent
- 明明只是分类问题，却设计了 planner
- 明明单 agent 足够，却为“高级感”上了 multi-agent

boundary checker 的价值是强迫你在开始编码前回答这些问题：
- 当前需求的不确定性到底在哪里
- 这些不确定性是入口不确定、步骤不确定，还是角色协作不确定
- 用更复杂的模式究竟能解决什么
- 如果不用更复杂的模式，是否已经足够

这本质上是在练“先减复杂度，再加能力”的习惯。

## 4. 为什么模式名越多，不代表系统越成熟
很多人写项目介绍时，喜欢说系统用了 Router、Planner、Supervisor、Memory、Evaluator，听起来很强。但模式名越多，常常只说明一件事：系统复杂度在上升。

真正成熟的系统，通常有这几个特征：
- 结构克制
- 状态边界清楚
- trace 可读
- 每一层的责任单一
- 失败可归因

换句话说，成熟不是“堆功能”，而是“你知道哪里不该加东西”。这也是为什么这个模块特别强调反例。学架构时，知道“不该怎么做”往往比知道“可以怎么做”更值钱。

## 5. 反例拆解：错误模式选择会怎么伤害系统
### 反例一：把 Router 问题做成 Planner 问题
假设你的系统只是要把问题分成：
- FAQ
- 订单查询
- 投诉工单

这本质上是入口分流问题。最合适的是 Router。如果你硬做成 Planner，系统会产生额外成本：
- planner 输出计划步骤
- executor 再执行计划
- trace 多出一层中间计划
- 错误从“分类错了”变成“计划错了还是执行错了”

收益几乎没有，排障成本却显著上升。

### 反例二：把 Workflow 问题做成自由 Agent 问题
如果人工审核节点、异常分支、状态转换都已经明确，workflow 最大的优势就是可预测。你把它换成自由 agent 后，常见后果是：
- 同一个输入不同次运行轨迹差异过大
- 审核点不稳定
- 回归测试变难
- 合规风险上升

### 反例三：把单上下文问题做成 multi-agent 问题
如果一个任务的核心上下文本来就高度耦合，硬拆多个 agent 可能只会导致：
- 重复传递上下文
- 角色间对齐成本高
- 信息丢失
- trace 断裂

这种情况下，multi-agent 不但没降复杂度，反而把原本清晰的问题拆散了。

## 6. 计划为什么值得显式化
很多人会问：既然模型也能边想边做，为什么还要显式 plan？

答案是：当计划本身对治理有价值时，显式化就值得。

显式计划通常带来这些收益：
- 可以人工审核
- 可以缓存或复用
- 可以从中间步骤恢复
- 可以对每一步单独打分
- 可以比较计划质量和执行质量是否一致

但计划也有代价：
- 增加一次生成
- 计划可能过度理想化，后面仍要修正
- 计划与执行之间可能出现偏离

所以不是所有系统都该先计划。关键判断是：计划作为对象，是否真的值得被治理。

## 7. architecture-lab 为什么要做成可运行系统
如果只是看文档，你很容易在脑子里觉得“这些模式差不多”。但一旦把它们放进同一个实验室里对比，你会清楚看到：
- 同样一个需求，Router 的状态最轻
- ReAct 的状态沿 observation 逐步累积
- Plan-and-Execute 多了显式计划对象
- Supervisor 多了协调和汇总层

这时差异不再是抽象概念，而是可以落到：
- 输入输出结构
- trace 形态
- 状态对象
- 失败传播路径

这也是为什么本模块不只讲理论，而一定要配 `architecture-lab`。因为真正的架构理解，必须落回可运行结构。

## 8. 如何用 trace 读出模式差异
如果你后面真的在系统里接 tracing，你会发现不同模式的 trace 读起来很不一样。

### Router 的 trace
- 开头通常有分类节点
- 后面迅速进入某条确定路径
- trace 较短，分支点较明确

### ReAct 的 trace
- 会出现多轮“思考/行动/观察”
- 每次 observation 都可能改变后续方向
- trace 更像递进式探索

### Plan-and-Execute 的 trace
- 前面会出现一个明确的 plan 生成阶段
- 中间每步执行通常能跟 plan 逐项对照
- 很适合做“计划偏差分析”

### Supervisor 的 trace
- 会出现顶层协调节点
- 下层多个执行分支并行或轮流被调度
- trace 阅读成本最高，但也最能体现协调逻辑

这意味着：模式不只是实现差异，也会直接决定你未来怎么 debug。

## 9. 为什么这一模块特别适合做“为什么不用另一个方案”的练习
很多工程师的问题不是不会做方案，而是不会拒绝方案。别人提一个“更智能”“更高级”的路线时，容易被说服。

这一模块真正训练的是反向判断能力：
- 为什么不用 multi-agent
- 为什么不用 planner
- 为什么不用更自由的 agent
- 为什么不用更复杂的 memory 结构

当你能稳定说出“不用某方案”的原因时，说明你已经开始在做架构，而不是被新概念牵着走。

## 10. 学完这一篇的完成标准
你应该能把一个需求用下面的方式拆开分析：
- 先判断它的核心复杂度在入口、执行中，还是角色协作上
- 再判断状态该集中还是拆分
- 再判断计划是否需要显式化
- 再判断 trace 和 eval 是否足以支撑更高复杂度
- 最后给出一个“为什么不用更复杂方案”的解释

如果你能做到这一点，这一模块就真的达成目标了。因为架构判断的成熟，往往不体现在你知道多少模式，而体现在你能把复杂度控制在刚刚好的位置。
